# modular-sc-daw

`modular-sc-daw` is a milestone-driven prototype modular DAW that is being migrated from a JUCE shell + SuperCollider core into a fully JUCE-native application.

## Current Migration State

- Legacy milestones `M1` through `M30` exist in prototype form.
- Migration milestones `J1`, `J2`, `J3`, `J4`, `J5`, `J6`, `J7`, and `J8` are now in place.
- The app no longer requires `sclang` or `scsynth` at runtime to launch.
- The current engine state shown by the UI is supplied by an internal JUCE compatibility layer.
- Global transport playback and ruler timing are now driven by a JUCE-native `TransportEngine`.
- Clock domains are now computed by a JUCE-native `ClockDomainManager`, including global, local, derived, and free-running demo domains.
- Modules are now authored by a JUCE-native `ModuleRegistry` with explicit demo module classes.
- Audio output is now generated by a JUCE-native `AudioEngine`.
- Mixer levels, mute state, master gain, and route enablement now affect the native audio path through JUCE-native mixer and routing layers.
- Quantised actions now resolve through a JUCE-native `Scheduler`.
- Editable behaviour surfaces now parse into native module behaviour and keep the last valid state on errors.

## Conceptual Model Being Preserved

The migration keeps the conceptual system intact while changing the implementation substrate:

- polymorphic modules
- capability-based module model
- global transport
- clock domains
- local / derived / free-running timing modes
- global ruler with local overlays
- routing
- mixer
- structural / conductor behaviour
- editable behaviour surfaces

## Migration Approach

This is not a blind rewrite.

The migration proceeds in explicit stages:

- `J1` remove runtime SuperCollider process dependency
- `J2` move transport authority into JUCE
- `J3` move clock-domain authority into JUCE
- `J4` move module runtime into JUCE
- `J5` move audio generation into JUCE
- `J6` move routing and mixer execution into JUCE
- `J7` move quantised scheduling into JUCE
- `J8` replace SC code surfaces with JUCE-native editable behaviours
- `J9` migrate persistence
- `J10` clean docs and retire legacy SC runtime files

## Current Runtime

Right now the app launches as a JUCE desktop application with:

- an internal engine status lifecycle: `offline -> booting -> ready`
- in-memory demo transport, clock, module, routing, mixer, automation, analysis, structural, recovery, render, and validation state
- no child-process launch of `sclang`
- no dependency on `sc-engine` at runtime

This keeps the UI usable while the deeper engine migration is completed in later `J` milestones.
